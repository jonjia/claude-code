# Claude Code 产品设计金矿清单

> 视角：从 **产品设计决策** 而非技术实现来看 Claude Code。
> 重点深挖：#5 AskUserQuestion · #6 dangerously 命名 · #8 Sub-agent · #9 双轨 Compaction

---

## 全景：3 条暗线

Claude Code 的产品设计有 3 条贯穿性的哲学：

1. **硬约束 > 软提醒** —— Plan mode、Permission mode、Worktree，都用机制（不是 prompt）保证安全
2. **把 "AI 自己的事" 对用户隐藏** —— Auto Memory、Compaction、Sub-agent 摘要，用户不需要管 agent 的内部状态
3. **复用用户已有习惯** —— CLAUDE.md（markdown）、Skills（slash 命令）、Worktree（git），不发明新格式

## 重点深挖

### #5 AskUserQuestion —— "agent 可以反问，但必须给选项"

**决策**：模型遇到歧义时主动反问用户，且 **必须带 2-4 个预设选项**（带 description）。强制要求不允许开放式提问。

**对比另一条路**：
- (A) agent 自作主张猜一个答案
- (B) agent 反问，但开放式："你想要什么？"

**为什么这条更好**：
- 比 (A) 好：避免 agent 走错方向，浪费整个回合
- 比 (B) 好：开放式问题让用户负担"翻译需求 → 文字"，选项把问题降维成"挑选"。**选项本身是认知脚手架**，告诉用户"这件事的合理选择空间是什么"

**深层洞察**：
> 选项不只是省力，它在 **教育用户什么是合理的选择空间**。开放式问题让用户用自己的词汇，往往词不达意；选项让用户在产品定义的语义空间里选择。

**可迁移场景**：
- AI 客服：别问"您还有什么问题"，给 3-4 个常见问题按钮
- 数据分析 agent：别问"你想分析什么"，给"趋势 / 对比 / 异常 / 分布"
- 设计 agent：别问"什么风格"，给具体风格预览图

**反模式警示**：很多 AI 产品做了反问，但用开放式 input 接收答案，结果用户回复短信式"嗯""可以""你看着办"，agent 又得追问，体验劣化。

### #6 `--dangerously-skip-permissions` —— "用命名摩擦替代弹窗摩擦"

**决策**：全自动跳过权限检查的参数命名为 `--dangerously-skip-permissions`，用户必须在命令行打出"dangerously"才能启用。

**对比另一条路**：
- (A) 叫 `--auto` 或 `--yes`（npm/yarn/apt 都这么命名）
- (B) 叫 `--no-confirm`（中性）
- (C) 第一次用时弹个"你确定吗"对话框

**为什么这条更好**：
- 比 (A)(B) 好：中性命名让用户低估风险，写进脚本/CI 后就忘了
- 比 (C) 好：弹窗会被点掉，且只警告一次

**深层洞察**：
> **摩擦应该放在"认知阶段"而不是"执行阶段"**。让用户在打字、复制、写文档、教别人的时候持续看到"dangerously"，比执行时弹一个 dialog 有效得多。这是 **永久摩擦 vs 一次性摩擦** 的差别。

**类似设计**：
- Linux `rm -rf /` 加 `--no-preserve-root` 才允许
- React 早期 `dangerouslySetInnerHTML`
- Stripe API key 测试/生产用 `sk_test_` / `sk_live_` 前缀

**可迁移场景**：
- 任何"破坏性操作的快速通道"，命名都该把后果写在脸上
- API 字段：删除接口加 `permanently_delete: true` 比 `force: true` 强
- 配置项：`disableSafetyChecks` 比 `safety: false` 强

**反模式警示**：用 `force: true` 这种通用词、用 `--auto` 模糊命名，都是把责任从产品转嫁给用户。

### #8 Sub-agent —— "保护主对话的可读性"

**决策**：把探索/搜索类的脏活派给 sub-agent（独立 context window），父 agent 只看到最终摘要，用户对话流里只显示一行"Agent: explored 12 files, found X"。

**对比另一条路**：
- (A) 所有工具调用都堆在主对话里
- (B) 折叠工具调用，但仍在主流里占位

**为什么这条更好**：
- 用户视角：对话流 = **决策链**，不是 **执行日志**
- agent 视角：主 context 不被几百屏 grep 结果淹没，剩下的 token 用来"思考"
- 产品视角：用户回看历史时能快速理解"我们做了什么"而不是"翻了多少文件"

**深层洞察**：
> **AI 产品的核心 UX 资源是"用户的注意力轨迹"**。Sub-agent 把噪音吞掉，等价于给主对话做了 "信噪比" 优化。这跟 IDE 的"折叠"不同——折叠仍要求用户决定要不要展开，而 sub-agent 是产品替用户决定了"这部分你不需要看"。

**何时该上 sub-agent**：
- 任务可以独立完成、产出能压缩成短摘要
- 中间过程对用户决策没价值（搜索、扫描、统计）
- 任务有明确边界（不需要持续追问用户）

**可迁移场景**：
- 数据分析 agent：跑 SQL 调试用 sub-agent，主对话只看结果
- 文档 agent：检索引用用 sub-agent，主对话只看引文
- 客服 agent：知识库查询用 sub-agent，主对话只看建议

**反模式警示**：很多 agent 产品炫耀"看我调了多少 tool"，把工具调用当成功能展示。这是**工程师视角**而非用户视角。用户要的是结果。

### #9 双轨 Compaction —— "自动是默认体验，手动是控制感"

**决策**：上下文压缩有两条路径
- **自动**：达到阈值时后台触发，用户无感
- **手动 `/compact`**：用户主动触发，可以指定保留什么

**对比另一条路**：
- (A) 只做自动 → 用户失去控制感，被截断时无能为力
- (B) 只做手动 → 普通用户根本不知道要压缩，对话越来越慢
- (C) 弹"对话太长，要压缩吗" → 弹窗疲劳

**为什么这条更好**：
> 这是 **AI 产品自动化设计的经典平衡**：默认路径覆盖 90% 用户，逃生口给 10% power user。**两个都不能少**。

**深层洞察**：

| 维度 | 自动 compaction | 手动 /compact |
|------|----------------|---------------|
| 服务对象 | 普通用户 | Power user |
| 触发时机 | 阈值被动 | 用户主动 |
| 心智负担 | 0 | 需要懂概念 |
| 控制粒度 | 粗（默认策略） | 细（可指定） |
| 可发现性 | 不需要发现 | 需要文档/教程 |

**关键设计**：自动版的策略要保守（多保留），手动版给激进选项（多压缩）。这样两者各自服务最佳场景。

**可迁移场景**：
- 笔记 AI：自动归档旧笔记 + 手动 /archive 命令
- 邮件 AI：自动分类 + 手动重分类
- 代码 AI：自动 import 整理 + 手动 /organize-imports
- 浏览器 AI：自动关闭闲置 tab + 手动 /close-old

**反模式警示**：
- 只做自动 → 用户投诉"为啥我的内容没了"
- 只做手动 → 用户投诉"为啥越用越卡，没人提醒我"
- 加弹窗 → 用户关掉所有提醒后再投诉以上两点

## 其余 7 项（参考索引）

### S 级（产品哲学，可迁移到任何 AI 产品）

- **#1 Plan Mode**：用一对工具做强制状态机，plan 期间所有写操作权限被拒。**信任靠流程门，不靠 prompt 提醒**。
- **#2 Permission Mode 分级**：4 档（default / acceptEdits / bypassPermissions / plan）给用户**渐进式信任曲线**，避免逐次弹窗疲劳。
- **#3 Auto Memory**：agent 自动判断"什么值得记"，无需用户 /save。**记忆是副产物，不是任务**。
- **#4 CLAUDE.md**：项目规则用 markdown 写，层级合并，支持 @import。**用用户最熟的格式表达 context，不发明新东西**。

### A 级（精巧，值得抄）

- **#7 Worktree 集成**：agent 改动放临时 git worktree，回滚做成默认基建而不是事后救火。

### B 级（场景化，但有启发）

- **#10 Skills（/teach-me 这种）**：把反复出现的 prompt 模板上升为命令，有命名、版本、参数。**prompt 工程产品化**。
- **#11 Hooks（PreToolUse / PostToolUse）**：让用户脚本拦截 tool 调用前后。**承认 agent 不完美，留逃生口**。

## 给做 AI 产品的人的 5 条带走

1. **反问要带选项**。开放式反问 = 把问题踢回给用户。选项 = 锚定理解 + 教育语义空间。
2. **危险操作的快速通道，要把后果写进命名**。`dangerously` 比 `force` 强，永久摩擦比弹窗有效。
3. **保护主对话的信噪比**。Sub-agent / 折叠 / 摘要，让用户看到决策链不是执行日志。
4. **自动化 + 控制感双轨**。默认路径服务普通用户，逃生口服务 power user，两个都不能少。
5. **硬约束 > 软提醒，复用习惯 > 发明新格式**。这两条是最容易被工程师忽略、又最影响产品质感的。
